# WESS 파일럿 — 백엔드/프론트엔드 통합 배포 가이드

이 문서는 새로 구현한 `backend`(Spring Boot, 일지/양식/MinIO/OnlyOffice 연동)와
`frontend`(React + Vite)를 기존 11.11.11.99 docker-compose 스택에 통합하는 절차를 정리한 것이다.
기존 구성은 `WESS_구성현황.md` 3절을 참고.

## 1. 변경 요약

- `backend/Dockerfile` 신규 추가 — Maven 멀티스테이지 빌드로 `app.jar` 생성 후 `eclipse-temurin:8-jre`에서 실행
- 루트 `docker-compose.yml` 갱신
  - `springboot-app`: placeholder(`eclipse-temurin:8-jdk` + jar 마운트) → `build: ./backend` 로 전환
  - `application.yml`의 환경변수(`SPRING_DATASOURCE_*`, `MINIO_*`, `ONLYOFFICE_INTERNAL_URL`, `BACKEND_INTERNAL_URL`) 전부 명시
  - `nginx`: `index.html` 단일 마운트 → `frontend/dist` 디렉터리 전체 마운트(SPA)
- `nginx.conf`에 SPA 라우팅(`try_files ... /index.html`) 추가 필요 (아래 4절)

## 2. 백엔드 빌드/배포

서버에 Maven/JDK를 설치하지 않고 Docker 빌드로 처리한다.

```bash
# 로컬 또는 서버에서 backend/ 디렉터리를 /data/pilot/backend 로 업로드(SFTP) 후
cd /data/pilot
docker compose build springboot-app
docker compose up -d springboot-app
```

- `pom.xml`의 `finalName=app` → 컨테이너 내부 `/app/app.jar`
- 기존처럼 jar 파일을 직접 SFTP로 올리는 방식(`mvnw clean package` → `app.jar` 교체 → `docker compose restart`)도 계속 가능하다. 이 경우 `docker-compose.yml`의 `springboot-app`에서 `build:` 대신 다시 `image: eclipse-temurin:8-jdk` + `volumes: - ./app.jar:/app/app.jar` + `command: java -jar /app/app.jar` 조합으로 되돌리면 된다.

### 필수 환경변수 점검

`docker-compose.yml`의 `springboot-app.environment`와 다음이 서로 일치해야 한다:

| 변수 | 값 | 일치해야 하는 대상 |
|---|---|---|
| `SPRING_DATASOURCE_PASSWORD` | `your_secure_password` | `postgres-db.environment.POSTGRES_PASSWORD` |
| `MINIO_ACCESS_KEY` | `admin_access_key` | `minio-storage.environment.MINIO_ROOT_USER` |
| `MINIO_SECRET_KEY` | `admin_secret_key` | `minio-storage.environment.MINIO_ROOT_PASSWORD` |
| `MINIO_BUCKET` | `wess-bucket` | 기존에 생성된 버킷명 (`mc mb local/wess-bucket` 완료됨) |

**운영 전환 시 `your_secure_password` 등 placeholder 값은 반드시 변경할 것.**

### 템플릿 docx 저장 경로

- `templates/{formTemplateId}/template.docx` — 관리자가 양식 등록 시 첨부
- `{year}/{semester}/{subject}/student_{studentId}/week_{n}/log_file.docx` — 학생 주차별 일지

두 경로 모두 `wess-bucket` 내부이며, 버킷은 외부 비공개(서버가 스트리밍 중계)이므로 별도 설정 불필요.

## 3. 프론트엔드 빌드/배포

```bash
cd frontend
npm install
npm run build   # → frontend/dist/
```

빌드 결과(`frontend/dist/`)를 서버의 `/data/pilot/frontend/dist`로 업로드한다.
`docker-compose.yml`의 `nginx` 서비스가 이를 `/usr/share/nginx/html`로 마운트한다.

> **이번 세션의 빌드 검증 관련 참고**: 이 작업 환경(sandbox)에서는 일부 파일(`AdminDashboard.tsx`, `StudentDashboard.tsx`, `SupervisorDashboard.tsx`, `types/index.ts`)에 대한 수정이 빌드용 마운트에 지연 동기화되는 문제가 있어, `npm run build`로 최종 빌드 산출물까지는 sandbox 내에서 직접 확인하지 못했다. 실제 파일 내용(에디터/Read 기준)은 task #17/#18 변경사항이 모두 반영된 최신 상태이며, 백엔드는 `mvn package -DskipTests`로 정상 빌드(`target/app.jar`, 48MB) 확인됨. **사용자 환경에서 `cd frontend && npm install && npm run build` 1회 실행으로 최종 확인을 권장**한다. (`tsc -b && vite build` 스크립트에서 `tsc -b`의 `vite.config.js` emit 관련 에러가 발생하면, 임시로 `tsc --noEmit && vite build`로 바꿔 빌드해도 된다.)

## 4. nginx.conf 추가 사항

기존 `/data/pilot/nginx.conf`의 `/api/`, `/minio/`, OnlyOffice 정적/동적 경로, `/example/` 라우팅은 그대로 유지한다.
정적 파일 서빙 부분(`location /`)만 SPA(React Router) 대응으로 아래와 같이 수정한다:

```nginx
location / {
    root /usr/share/nginx/html;
    try_files $uri $uri/ /index.html;
}
```

- 반드시 `/api/`, `/minio/`, OnlyOffice 관련 `location` 블록보다 **아래**(우선순위 낮은 위치)에 두어, 해당 경로들이 먼저 매칭되도록 한다.
- 기존에는 `index.html` 한 파일만 마운트했지만, 이제 `frontend/dist` 전체(`index.html`, `assets/` 등)를 마운트하므로 정적 자산(JS/CSS) 경로가 `/assets/...`로 정상 서빙되는지 확인 필요.

## 5. 배포 절차 요약

1. `backend/`, `frontend/dist/`, 갱신된 `docker-compose.yml`, `nginx.conf`를 `/data/pilot/`에 SFTP 업로드
2. `cd /data/pilot && docker compose build springboot-app`
3. `docker compose up -d`
4. 헬스체크: `docker compose ps` 5개 컨테이너 Up, `https://11.11.11.99/api/...` 응답 확인
5. 로그인 → 학생/지도자/관리자 대시보드에서 양식 등록 → 일지 작성/저장 → 지도자 조회 → 관리자 PDF 일괄 다운로드 흐름 점검

## 6. 향후 정리할 항목

- `your_secure_password`, MinIO access/secret key 등 placeholder를 운영용 값으로 교체
- `JWT_SECRET`(OnlyOffice) 운영 시 별도 관리 방안 검토
- SSL은 현재 self-signed — 운영 전환 시 정식 인증서로 교체
